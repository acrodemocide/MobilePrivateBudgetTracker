DATABASE RECOMMENDATION FOR MOBILEPRIVATEBUDGETTRACKER
=======================================================

SQLite is the industry standard for mobile app databases, and the right choice here.

Both iOS and Android ship SQLite built into the OS -- it's what Apple's Core Data and
Google's Room are built on top of. Virtually every major mobile app that stores structured
local data (banking apps, note-taking apps, etc.) uses SQLite under the hood.

WHY IT'S THE RIGHT FIT FOR A BUDGET TRACKER SPECIFICALLY:
- Transactions are structured, relational data -- exactly what a SQL database is designed for
- You'll eventually want to query by date range, filter by category, sum amounts, etc.
  -- all trivially easy with SQL
- It handles thousands or millions of rows with no performance concerns
- Data is stored in a single file on the device, survives app restarts and updates

RECOMMENDED LIBRARY:
@op-engineering/op-sqlite
  - Current best-practice choice for React Native + SQLite
  - Faster than the older react-native-sqlite-storage
  - Actively maintained, written in C++ for performance

ALTERNATIVES AND WHY THEY'RE WEAKER FOR THIS USE CASE:
- AsyncStorage   : Simple key-value settings. No querying; storing a JSON blob of all
                   transactions gets unwieldy fast.
- MMKV           : Fast key-value (e.g., user prefs). Same problem as AsyncStorage.
- Realm          : Heavy dependency; overkill for transactions.
- WatermelonDB   : Very large datasets with sync. Significant complexity overhead for
                   this scale.
